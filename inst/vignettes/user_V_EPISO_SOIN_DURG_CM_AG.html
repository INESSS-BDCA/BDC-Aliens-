1 Installation de la librairie R RequeteGeneriqueBDCA

1.1 Fichier nécessaire

Télécharger le fichier RequeteGeneriqueBDCA.x.y.tar.gz à partir de Microsoft Teams.
INESSS - Documentation BDCA > Documents > Outils > Librairie R INESSS > RequeteGeneriqueBDCA_x.y.z.tar.gz.

Ouvrir RStudio.

Dans la console, écrire le code suivant :

remotes::install_local("C:\\msXXX\\Desktop\\RequeteGeneriqueBDCA_x.y.z.tar.gz")

Prendre note que les «\» sont doubles.

Si la librairie remotes n’est pas installée, écrire dans la console le code suivant : install.packages("remotes").

Avant d’installer une librairie, il est préférable de ne pas l’avoir appelé auparavant.

Il est déconseillé de mettre à jour les librairies utilisées par RequeteGeneriqueBDCA.

L’installation se termine lorsqu’on peut lire dans la console : DONE (RequeteGeneriqueBDCA).

1.2 Activer la librairie

Après avoir installé la librairie RequeteGeneriqueBDCA, il suffit de l’activer pour avoir accès aux fonctions :

library(RequeteGeneriqueBDCA)

2 user_V_EPISO_SOIN_DURG_CM_AG

La fonction user_V_EPISO_SOIN_DURG_CM_AG est destinée aux utilisateurs. Elle permet:

  1. d’extraire les variables pertinentes de la base de données BDCU V_EPISO_SOIN_DURG_CM, jumlées avec d’autres bases de données pour rajouter les caractéristiques du bénéficiaire et de l’établissement de soins.
  2. d’extraire les variables pertinentes de la base de données BDCU V_OCCU_CIVIE_DURG_CM.
  3. d’extraire les variables pertinentes de la base de données BDCU V_CNSUL_DURG_CM.

La fonction user_V_EPISO_SOIN_DURG_CM_AG est une child_function qui permet d’exécuter les requêtes SQL générées par les parent_function suivantes:
SQL_reperage_cond_med
query_V_EPISO_SOIN_DURG_CM_AG
query_V_FICH_ID_BEN_CM

Si la cohorte n’est pas fournie par le professionnel scientifique, et on veut créer une cohorte en utilisant les Dx cim-9 et/ou cim-10:

la fonction user_V_EPISO_SOIN_DURG_CM_AG exécute trois étapes principales:
1) la création de la cohorte de l’étude en utilisant les Dx cim-9 et/ou cim-10 via la fonction SQL_reperage_cond_med
2) l’extraction des variables pertinentes pour cette cohorte via la fonction query_V_EPISO_SOIN_DURG_CM_AG
3) l’ajout des caractéristiques de benéficiaire, ainsi que de l’établissement de soins.

Si la cohorte n’est pas fournie par le professionnel scientifique, et on ne veut pas créer une cohorte en utilisant les Dx cim-9 et/ou cim-10:
l’étape 1 est ignorée et les étapes 2 et 3 sont exécutées en fonction de la nature de l’argument task

2.1 La fonction


user_V_EPISO_SOIN_DURG_CM_AG(task,
                          conn=SQL_connexion(),
                          cohort=NULL, # si la cohorte est fournie par l'analyste (cohort=cohort), si on veut créer une nouvelle cohorte (cohort=NULL)
                          debut_periode,
                          fin_periode,
                          benef_adr,
                          date_adr, # à spécifier lorsque benef_adr="date_fixe"
                          date_age, # à spécifier lorsque keep_all=TRUE
                          diagn, # liste de Dx pour la task: extraction_BDCU_EPISO_SOIN_DURG 
                          
                          Dx_table, # si la cohorte n'est pas fournie par le professionnel scientifique et nous ne voulons pas créer une nouvelle cohorte (Dx_table=NULL)
                          debut_cohort,
                          fin_cohort,
                          CIM,
                          by_Dx,
                          date_dx_var,
                          n1,n2,
                          nDx,
                          keep_all=FALSE,
                          
                          verbose=TRUE,
                          setwd)

2.2 Définition des arguments

2.2.1 Connexion

La fonction SQL_connexion permet à R de se connecter à Teradata. Simplenent inscrire l’identifant et le programme demandera le mot de passe lors de son exécution :

conn = SQL_connexion("msXXX")

2.2.2 cohorte

Vecteur indiquant les numéros d’identification des bénéficiaires définisant la cohorte de l’étude.

cohort<-cohort$BENF_NO_INDIV_BEN_BANLS

2.2.3 Période de l’étude

Date de début et de fin de la période d’étude au format AAAA-MM-JJ.

debut_periode = "2020-01-01" 
fin_periode = "2020-12-31"

2.2.4 Paramètres spécifiques de la requête

2.2.5 Liste des diagnostics

list contenant les codes diagnostics CIM9 et CIM10.

diagn = list(
  diabete = list(
    CIM9 = c("2500%", "2501%", "2502%"),
    CIM10 = c("E100%", "E101%", "E109%", "E110%", "E111%", "E119%", "E130%",
              "E131%", "E139%", "E140%", "E141%", "E149%")),
  diabete_complication = list(
    CIM9 = paste0(2503:2509, "%"),
    CIM10 = paste0("E", c(102:108, 112:118, 132:138, 142:148), "%")
  )
)

2.2.6 Caractéristiques du bénéficiaires

2.2.6.1 Adresse du bénéficiaire

Permet de définir la méthode d’identification de l’adresse de bénéficiaire.

benef_adr=c("date_fixe","dernière_adresse","première_adresse","long_adresse")

2.2.6.2 Date adresse

Si benef_adr=“date_fixe”, date_adr permet de choisir une date fixe pour identifier l’adresse de bénéficiaire.

date_adr="2020-07-01"

2.2.6.3 Date âge

Permet d’indiquer une date "AAAA-MM-JJ" à laquelle l’âge des bénéficaires qui n’ont pas eu d’acte est calculé. L’âge de bénéficiaires qui ont eu un acte à l’intérieur de la période de l’étude est calculé à la date de l’acte.

date_age="2020-03-01"

2.2.7 PARAMÈTRES POUR LA CRÉATION DE LA COHORTE AVEC SQL_reperage_cond_med

2.2.7.1 Liste des diagnostics

list contenant les codes diagnostics CIM9 et CIM10.

Dx_table = list(
  diabete = list(
    CIM9 = c("2500%", "2501%", "2502%"),
    CIM10 = c("E100%", "E101%", "E109%", "E110%", "E111%", "E119%", "E130%",
              "E131%", "E139%", "E140%", "E141%", "E149%")),
  diabete_complication = list(
    CIM9 = paste0(2503:2509, "%"),
    CIM10 = paste0("E", c(102:108, 112:118, 132:138, 142:148), "%")
  )
)

2.2.7.2 Classification des diagnostics

Permet de choisir le type de code diagnostic : CIM9, CIM10 ou les deux.

CIM = c("CIM9", "CIM10")  # par défaut : CIM9 et CIM10
CIM = "CIM9"  # CIM9 seulement
CIM = "CIM10"  # CIM10 seulement

2.2.7.3 Date début et fin de la cohorte

Date de début et de fin de la création d’une cohort au format AAAA-MM-JJ.

debut_cohort = "2018-01-01" 
fin_cohort = "2020-12-31"

2.2.7.4 Stratification de la cohorte

Vrai (TRUE) ou FAUX (FALSE).

À partir de DX_table créé précédemment, si TRUE, on aurait au maximum deux lignes par individu où la colonne DIAGN indiquerait diabete ou diabete_complication.
Si FALSE, on a au maximum une ligne par individu et la colonne DIAGN est absente, car les codes de diabete et diabete_complication sont considérés comme un seul élément d’analyse dans l’algorithme.

by_Dx = TRUE  # par défaut
by_Dx = FALSE

2.2.7.5 Date d’admission ou de départ

Permet de choisir entre la date d’admission ou la date de départ comme date d’incidence dans les vues suivantes :

  • V_DIAGN_SEJ_HOSP_CM
  • V_SEJ_SERV_HOSP_CM
  • V_EPISO_SOIN_DURG_CM
date_dx_var = "admis"  # par défaut
date_dx_var = "depar"

2.2.8 Paramètres de l’algorithme de création de cohorte

Permet d’indiquer les paramètres de l’algorithme soit : le nombre de jours (n1) entre 2 diagnostics, la période de temps (n2) à considérer pour confirmer le premier diagnostic et le nombre minimum d’occurrences du diagnostic pour confirmer la condition (nDx). (Valeurs par défaut n1= 30 jours, n2= 730 jours et nDx=1)

2.2.8.1 Nombre de jours (n1 & n2)

Permet de créer l’intervalle [n1, n2]. Le nombre de jours entre deux diagnostics doit se situer dans cet intervalle pour que le deuxième diagnostic confirme le premier.

n1 = 30 
n2 = 730

2.2.8.2 Nombre d’occurence (nDx)

Permet de choisir le nombre de diagnostic qui permet de confirmer le premier.

nDX=1 # par défaut

2.2.9 Cas retenus

2.2.9.1 keep_all

TRUE ou FALSE. Par défaut FALSE, soit filter les observations ayant eu un DX et un code d’acte. Si TRUE garder toutes les observations, soit ceux qui ont eu un DX et qui ont eu ou pas un code d’acte. Si cohort est fournit, keep_all TRUE garde toutes les observations demandées par cohort. Si keep_all FALSE filtrer les observations demandées par cohortqui ont eu un code d’acte.

2.2.10 setwd

Permet d’indiquer le chemin vers le dossier dans lequel les résultats de la fonction seront sauvegardés. Si setwd=“NULL”, les résultats seront seulement affichés dans l’environnement du Rstudio.

setwd="V:/GI/Projets/4.Projets_en_cours/Nom_dossier"

2.2.11 verbose

TRUE ou FALSE. Message de progression affiché dans la console.

Étape 1: Création d’une cohorte en utilisant les Dx cim-9 et/ou cim-10: nDx=0 : Extraction des résultats (Pas de confirmation nécessaire). V_DIAGN_SEJ_HOSP_CM - Coronaro (4.93 secs) V_SEJ_SERV_HOSP_CM - Coronaro (1.78 secs) V_EPISO_SOIN_DURG_CM - Coronaro (3.42 secs) I_SMOD_SERV_MD_CM - Coronaro (15.6 secs) Arrangement de la table cohorte… fin de création de la cohorte.

Étape 2: Extraction des variables BDCU pour la cohorte crée à l’étape 1: - Temps d’exécution (5.71 mins)

Étape 3: Préparation de la base de données finale: Garder les IDs de la cohorte crée à l’étape 1 qui ont eu recours aux services d’urgence à l’intérieur de la période de l’étude - Temps d’exécution (0.2 secs)

3 Table Résultante

3.1 task="extraction_BDCU_EPISO_SOIN_DURG"

Extraction des variables pertinentes de BDCU: V_EPISO_SOIN_DURG_CM combinées avec les caractéristiques du bénéf et de l’établissement. Une base de données avec 33 variables. - ID : Numéro du bénéficiaire
- SURG_NO_EPISO_SOIN_DURG : Numéro d’épisode de soins - Départements d’urgence
- SURG_AN_PER_TEMPS : Année période temps
- SURG_DHD_EPISO_SOIN_DURG : Date et heure de début d’un épisode de soins
- DD_SOIN_DURG : Date_debut_epi_soins
- SURG_DH_DEPAR_USAG_DURG : Date et heure de départ d’un usager
- DF_SOIN_DURG : Date_fin_epi_soins
- SURG_DH_PREM_TRIAG_DURG : Date et heure du premier triage d’urgence
- SURG_DH_PRIS_CHARG_DURG : Date et heure de prise en charge
- SURG_DH_DEM_ADMIS_DURG : Date et heure de la demande d’admission en CH
- SURG_DH_DECES_DURG : Date et heure du décès
- SURG_COD_PRIOR_TRIAG_DURG : Code de priorité du premier triage
- SURG_COD_AUTNM_PREM_TRIAG_DURG Code d’autonomie après le premier triage
- SURG_COD_RAIS_VISIT_DURG : Code de raison de visite en urgence
- SURG_CATG_MAJR_DIAGN_DURG : Catégorie majeure de diagnostic
- SURG_COD_DIAGN_MDCAL_CLINQ Code de diagnostic médical clinique
- DATE_DX : Définie selon SURG_DHD_EPISO_SOIN_DURG (admis) ou SURG_DH_DEPAR_USAG_DURG (depar)
- SURG_NBR_CNSUL_DURG : Nombre de consultations en département d’urgence
- SURG_COD_MODE_ARRIV_DURG : Code du mode d’arrivée de l’usager en urgence
- SURG_TYP_ORITN_USAG_DEPAR_DURG : Type d’orientation de l’usager à son départ
- SURG_NO_ETAB_MSSS : Numero de l’etablissement attribue par MSSS
- Sexe : Sexe du bénéficiaire
- DatNais : Date de naissance du bénéficiaire au format AAAA-MM-JJ
- DatDeces : Date de décès du bénéficiaire au format AAAA-MM-JJ
- RSS_Benf : Régions sociosanitaires (RSS) du bénéficiaire
- RLS_Benef : Réseaux locaux de services (RLS) du bénéficiaire
- NomRSS_Benef : Nom régions sociosanitaires (RSS) du bénéficiaire
- NomRLS_Benef : Nom réseaux locaux de services (RLS) du bénéficiaire
- RLS_ETAB : Régions sociosanitaires (RSS) de l’établissement
- NomRLS_Etab : Nom réseaux locaux de services (RLS) de l’établissement
- RSS_ETAB : Régions sociosanitaires (RSS) de l’établissement
- NOM_RSS_Etab : Nom régions sociosanitaires (RSS) de l’établissement
- NOM_ETAB : Nom de l’établissement

3.2 task="extraction_BDCU_OCCU_CIVIE_DURG"

Extraction des variables pertinentes de BDCU: V_OCCU_CIVIE_DURG_CM. Une base de données avec 8 variables. - ID : Numéro du bénéficiaire
- SURG_NO_EPISO_SOIN_DURG : Numéro d’épisode de soins - Départements d’urgence
- SURG_NO_SEQ_OCCU_CIVIE_DURG : Numéro de séquence de l’occupation de civière
- SURG_DHD_OCCU_CIVIE_DURG : Date et heure de début d’occupation de civière
- DD_Occup_civ : Date de début d’occupation de civière
- SURG_DHF_OCCU_CIVIE_DURG : Date et heure de fin d’occupation de civière
- DF_Occup_civ : Date de fin d’occupation de civière
- SURG_CATG_CIVIE_DURG : Catégorie de civière en département d’urgence

3.3 task="extraction_BDCU_CNSUL_DURG"

Extraction des variables pertinentes de BDCU: V_CNSUL_DURG_CM. Une base de données avec 9 variables. - ID : Numéro du bénéficiaire
- SURG_NO_EPISO_SOIN_DURG : Numéro d’épisode de soins - Départements d’urgence
- SURG_NO_SEQ_CNSUL_DURG : Numéro de séquence de consultation
- SURG_COD_SPEC_CNSUL_DURG : Code de spécialité de consultation d’urgence
- SURG_COD_ETA_CNSUL_DURG : Code spécifiant l’état de la consultation
- SURG_DH_DEM_CNSUL_DURG : Date et heure de la demande de consultation
- DD_DEM_CNSUL : Date de la demande de consultation
- SURG_DH_REAL_CNSUL_DURG : Date et heure de réalisation de consultation <br/ - DD_REAL_CNSUL : Date de réalisation de consultation
